# MarkBuddy 产品定位设计

日期：2026-06-30

## 摘要

MarkBuddy 应定位为浏览器阅读捕获层，而不是完整知识库或笔记软件。它的核心任务是帮助用户快速保存网页中的有用片段，并在之后找回原网页、原句和上下文。间隔复习有价值，但应保持主动选择和辅助定位。Markdown 导出是必要的反信息孤岛能力，让用户可以把高亮内容带到 Obsidian、Notion、Logseq、Git 仓库或普通文件夹中继续使用。

## 目标用户

MarkBuddy 聚焦两类有重叠的用户：

1. 经常阅读技术文档、教程、博客、issue、API 参考、StackOverflow 答案和实现方案的技术读者。
2. 经常阅读文章、 newsletter、长文、博客和观点分析，并希望保留少数有用或有记忆点段落的日常网页读者。

这些用户主要不缺另一个知识库。他们需要的是一种低摩擦方式，在不离开浏览器的情况下保存重要网页片段。

## 产品判断

产品应该优化这个结果：

> 我记得自己曾经在某处看到过一句有用的话。MarkBuddy 帮我找到它，回到原网页，并在需要时复习它。

这个定位比“书签管理器”或“知识管理工具”更清晰。MarkBuddy 保存的不是一个网页 URL，而是这个网页当时为什么值得保存。

## 核心优先级

### 1. 找回优先

主要工作流是找回曾经保存过的网页知识：

- 保存网页选中文本，并记录标题、URL、时间、高亮颜色、标签和可选备注。
- 用户再次访问网页时恢复高亮。
- 支持搜索网页标题、URL、高亮原文和备注。
- 支持按标签和页面分组过滤。
- 在可行时从侧边栏跳回原网页中的高亮位置。

这应该是 A/C 类用户的主价值，因为他们的共同痛点通常是找回，而不是正式学习。

### 2. 复习辅助

间隔复习应有用但克制：

- 只有带有配置复习标签的高亮进入复习队列。
- 复习应主动选择，而不是默认作用于所有高亮。
- 复习系统应帮助用户记住少数真正重要的摘录，而不是把所有阅读都变成作业。
- 复习体验应保留足够的原文和来源上下文，避免变成脱离语境的闪卡负担。

现有 SM-2 方向适合这个角色，前提是队列准入始终由用户主动控制。

### 3. Markdown 导出作为数据自由

Markdown 导出应被视为数据流转能力，而不是笔记编辑器替代品：

- 支持导出选中页面、全部页面，或当前搜索/标签过滤结果。
- 保留页面元信息、来源 URL、标签、高亮、备注和导出时间。
- 生成可读的 Markdown，方便进入常见下游工具。
- 避免绑定特定笔记软件，优先使用简单标题、引用块、元信息行和列表。

导出能力可以降低用户采用成本：使用 MarkBuddy 不意味着数据会被困在浏览器扩展里。

## 非目标

MarkBuddy 不应尝试变成：

- 完整笔记编辑器。
- Notion 或 Obsidian 替代品。
- 图谱知识库。
- 项目管理或任务系统。
- 社交阅读或分享平台。
- 对所有捕获内容强制生效的间隔复习系统。
- 复杂文件夹层级管理器。

这些方向会削弱产品，让它进入成熟且拥挤的品类，而浏览器扩展在这些战场上的优势并不明显。

## 推荐交互方向

侧边栏应表达三个实用模式，但不应变成三个割裂产品：

- 资料库：默认视图，用于搜索、标签、页面分组和高亮浏览。
- 复习：只有存在到期复习项时才突出展示的主动模式。
- 导出：可从选中页面、过滤结果或全部数据触发的工具动作。

导出应像一个可靠出口，而不是打断日常捕获和找回流程的主路径。

## Markdown 导出形态

单页导出：

```md
# 页面标题

- URL: https://example.com/article
- Saved: 2026-06-30
- Tags: tech, react, performance

## Highlights

### 1. Highlight

> 用户划线的原文。

Note: 用户备注。

Source: https://example.com/article
```

批量导出：

```md
# MarkBuddy Export

Exported: 2026-06-30

## example.com

### 页面标题

- URL: https://example.com/article
- Saved: 2026-06-30
- Tags: tech, react

#### Highlights

> 第一条高亮。

Note: 可选备注。

> 第二条高亮。
```

## 架构影响

现有扩展架构可以支撑这个方向：

- `service-worker.js` 继续作为书签、高亮、设置、标签、复习状态和后续导出数据组装的数据权威。
- `side-panel/panel.js` 继续作为资料库、复习和导出的主要用户界面控制层。
- `content/content.js` 继续聚焦网页捕获、高亮恢复和跳回原文。
- 导出逻辑应隔离为小型格式化模块或函数，方便在不依赖 DOM 渲染的情况下测试 Markdown 生成。

这个方向不需要新增后端、账号系统、同步服务或外部笔记服务。

## 数据流

捕获流程：

1. 用户在网页中划选文本。
2. 内容脚本序列化选区，并发送给 service worker。
3. Service worker 保存高亮，并确保该页面存在对应书签。
4. 侧边栏重新加载并展示页面和高亮。

找回流程：

1. 用户在侧边栏中搜索或过滤。
2. 面板在本地书签、URL、高亮和备注字段中检索。
3. 用户打开页面或跳转到某条高亮。
4. 内容脚本在可行时恢复或聚焦原始高亮。

复习流程：

1. 用户给选定高亮或页面打上配置的复习标签。
2. Service worker 返回到期高亮。
3. 面板展示复习卡片，并记录 SM-2 反馈。
4. Service worker 更新下一次复习时间。

导出流程：

1. 用户选择导出范围。
2. 面板请求或派生相关书签和高亮。
3. 导出格式化器将数据转换为 Markdown。
4. 浏览器下载 API 将 `.md` 文件保存到本地。

## 错误处理

找回应优雅处理网页变化：

- 如果高亮恢复失败，保留已保存数据，并将该高亮显示为失效或仅来源可用。
- 如果跳回原文失败，仍然打开页面 URL，并显示清晰的失败提示。
- 如果页面已经不存在，仍然保留已保存的高亮和元信息。

导出应保持宽容：

- 缺失标题时回退到 URL。
- 缺失标签时省略标签行。
- 缺失备注时省略备注行。
- 用户内容中的 Markdown 特殊字符应保持可读，并且不应破坏文档结构。

## 测试策略

测试应聚焦产品风险较高的部分：

- 搜索可以命中标题、URL、高亮原文和备注。
- 复习队列只包含匹配配置复习标签的高亮。
- Markdown 导出保留元信息、高亮、备注、标签和来源 URL。
- 单页、过滤结果和全部数据导出都可用。
- 空导出范围显示有用提示，而不是下载空文件。
- 高亮恢复和跳回原文失败时不会删除用户数据。

## 成功标准

MarkBuddy 在这个方向上成功，意味着：

- 用户可以用一到两个动作保存有用网页片段。
- 用户之后可以在不记得具体网页的情况下找到该片段。
- 当原网页仍可访问时，用户可以回到原始来源。
- 用户可以把少量重要高亮主动加入复习。
- 用户可以把高亮导出为 Markdown，并在其他地方继续使用。

## 决策

采用“找回优先、复习辅助、Markdown 导出保障数据自由”的方向。

这个方向能让 MarkBuddy 避开成熟知识库软件的正面竞争，同时解决一个具体的浏览器原生问题：保存和找回有用网页片段，并且不困住用户数据。
